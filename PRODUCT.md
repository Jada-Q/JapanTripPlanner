# Japan Trip Planner - 产品说明书

## 一、产品概述

**Japan Trip Planner** 是一款面向来日旅游者的智能行程规划工具，核心解决"信息碎片化、规划低效"的痛点。用户只需输入旅行日期、目的城市和个人偏好，即可一键生成带完整交通衔接的每日行程方案，并自动计算 JR Pass 是否值得购买。

### 核心定位

> 从"内容浏览"到"行动方案"——区别于小红书等 UGC 平台的碎片化攻略，提供结构化、可直接执行的旅行计划。

### 目标用户

- 首次赴日旅游、对日本交通系统不熟悉的游客
- 需要多城市连游、涉及新干线等复杂交通的旅行者
- 希望快速获得专业级行程方案的自由行用户
- 中文及英文用户群体

## 二、核心功能

### 1. 智能行程生成

- **输入**: 出发/结束日期、目的城市（支持多城市）、预算等级、旅行节奏、兴趣标签
- **输出**: 按天拆分的完整行程，每天包含景点活动、餐饮推荐、交通衔接
- **AI 引擎**: Google Gemini API，通过结构化 JSON 输出保证数据一致性

### 2. 交通深度整合

- 每段交通显示：出发站/到达站（含日文站名）、列车名称、车型、出发/到达时间、票价
- JR Pass 适用性标识——一眼看出哪些线路被 JR Pass 覆盖
- 站台提示和换乘注意事项

### 3. JR Pass 智能推荐

- 确定性算法（非 AI 猜测）：汇总行程中所有 JR 可用线路的票价，与全国 JR Pass 价格对比
- 支持 7 天 / 14 天 / 21 天普通席和绿色车厢的成本对比
- 明确给出"买 Pass 省多少"或"不需要买 Pass"的结论

### 4. 季节感知

- 樱花季（3-4月）、红叶季（10-12月）的最佳观赏地推荐
- 重大节庆和祭典提醒（祇园祭、天神祭等）
- 旅行警示期提醒（黄金周、盂兰盆节等高峰期）

### 5. 日文地址一键复制

- 每个景点显示日文地址，点击即复制到剪贴板
- 专为"给出租车司机看地址"的场景设计

### 6. 中英双语

- 完整的 UI 中英文切换
- 行程内容根据用户语言偏好生成对应语言版本

## 三、技术架构

```
┌─────────────────────────────────────────┐
│              Frontend (React)           │
│  Vite + TypeScript + Tailwind CSS       │
│  react-i18next + TanStack Query         │
├─────────────────────────────────────────┤
│              REST API (FastAPI)         │
│  /api/v1/itinerary/generate   POST     │

│  /api/v1/reference/jr-passes  GET      │
├───────────────┬─────────────────────────┤
│ Gemini API    │  Deterministic Engine   │
│ (行程生成)     │  (JR Pass 计算)         │
├───────────────┴─────────────────────────┤

```

### 技术选型理由

| 组件 | 选择 | 理由 |
|------|------|------|
| AI | Gemini API (JSON Mode) | 结构化输出、成本低、响应快 |
| 后端 | FastAPI (Python) | 异步支持、自动文档、Pydantic 数据验证 |
| 前端 | React + Vite + Tailwind | 开发效率高、生态成熟、样式灵活 |
| JR Pass | 确定性算法 | 不依赖 AI 猜测，结果可验证、可测试 |

## 六、未来发展路线图

### Phase 2: 增强体验（近期）

- **区域 JR Pass 优化**: 除全国 Pass 外，推荐关西/九州/北海道等区域 Pass，进一步节省用户交通费用
- **行程编辑与拖拽排序**: 允许用户手动调整生成的行程，增减景点、变更顺序
- **Streaming 响应**: 行程生成过程实时显示，减少用户等待焦虑
- **PDF/图片导出**: 一键导出行程为 PDF 或长图，方便离线查看和分享

### Phase 3: 用户系统与社交（中期）

- **用户账号系统**: 注册/登录，保存和管理多个行程
- **行程分享**: 生成分享链接，朋友可查看或复制行程
- **社区功能**: 用户发布实际旅行反馈，为景点和线路贡献评分
- **协作规划**: 多人共同编辑同一行程（适合团体旅行）

### Phase 4: 深度整合（中远期）

- **酒店/住宿推荐**: 根据每日行程终点推荐附近酒店，直连预订平台
- **实时列车时刻**: 对接 JR 等官方 API，提供实时车次信息
- **地图集成**: Google Maps / Apple Maps 嵌入，可视化每日路线
- **航班信息**: 根据出发城市推荐航班，计算机场到市区交通
- **餐厅预订**: 对接 Tabelog/Hot Pepper 等平台，支持预约热门餐厅

### Phase 5: 商业化与平台化（远期）

- **Freemium 模式**: 基础行程免费，高级功能（详细交通时刻、酒店比价、离线地图）付费
- **B2B API**: 向旅行社、OTA 平台提供行程生成 API
- **广告/联盟营销**: 酒店、交通、景点门票的 affiliate 收入
- **多国扩展**: 从日本扩展到韩国、东南亚等热门旅游目的地
- **旅行保险整合**: 推荐并销售旅行保险产品

### 技术演进方向

- **多模态 AI**: 支持用户上传图片（"我想去这个地方"）识别景点并加入行程
- **对话式规划**: 从表单输入升级为自然语言对话（"帮我规划一个带小孩的东京5日游"）
- **离线 PWA**: 渐进式 Web 应用，旅途中无网络也能查看行程
- **个性化学习**: 基于用户历史行程和反馈，持续优化推荐算法
- **多语言扩展**: 在中英文基础上增加日语、韩语、泰语等语言支持

## 七、市场机会

### 市场规模

- 2024年访日外国游客突破 3,600 万人次，其中中国游客占比最大
- 日本政府目标 2030 年实现年 6,000 万外国游客
- 自由行比例逐年上升，对规划工具的需求持续增长

### 竞争优势

1. **创始人地理优势**: 团队在东京，对日本交通、文化、季节有第一手认知

3. **数据积累**: 随着用户增长，行程数据和反馈将形成独有数据资产
4. **垂直深耕**: 聚焦日本一个市场做深，而非泛旅行平台的浅覆盖

### 盈利模式前景

| 阶段 | 收入来源 | 预期占比 |

| 早期 | 免费增长，积累用户 | — |
| 成长期 | Premium 订阅 + 酒店/交通 affiliate | 60% + 30% |

## 九、变更日志与经验教训

### v1.3 (2026-03-16) — 景点照片修复 + 数据健壮性

**变更内容：**
- 修复景点照片无法显示的 bug：AI 生成的 Wikipedia Commons 图片 URL 全部是虚构的（404）
- 改用 Wikipedia REST API 获取真实缩略图：AI 仅提供 `wikipedia_slug`（维基百科文章标题），后端通过 `https://en.wikipedia.org/api/rest_v1/page/summary/{slug}` 获取真实的 CDN 图片 URL
- 新增 `wikipedia_service.py`：并发解析所有景点的 Wikipedia 缩略图，使用 semaphore 限制并发数
- 添加 `httpx` 依赖用于高性能异步 HTTP 请求
- 前端迁移至 Next.js + shadcn/ui，使用 Route Handler 代理后端 API（解决 rewrites 超时问题）
- **全面防护 AI 返回数据**：为 Activity、DayPlan 的所有必填字段添加 setdefault 兜底，防止 AI 省略任何字段导致 Pydantic 验证失败

**经验教训：**
- **🚨 绝不让 AI 直接生成 URL**: AI 模型（包括 Kimi、Gemini、GPT）会"幻觉"出看似合理但实际不存在的 URL。即使明确要求提供 Wikipedia Commons 链接，AI 生成的 URL 也几乎全部返回 404。**根本原因**: LLM 不是搜索引擎，它不能实时验证 URL 是否存在。**解决方案**: 让 AI 只提供"标识符"（如 Wikipedia 文章标题），由后端代码调用真实 API 获取 URL。**通用原则**: 任何需要真实外部资源（图片、链接、API endpoint）的场景，都应由确定性代码获取，不能信任 AI 的输出。
- **并发请求需要节制**: 一次行程可能有 20+ 个景点，同时发起 20 个 Wikipedia API 请求可能触发限流。使用 `asyncio.Semaphore(10)` 限制并发数，兼顾速度和礼貌。
- **🚨 AI 返回的 JSON 字段不可靠——必须为所有字段设默认值**: AI 有时会省略 JSON schema 中的必填字段（如 `name_ja`、`address_ja`、`transport_segments`），或将必填字段设为 null。Pydantic 严格验证会直接报错导致 500。**解决方案**: 在 Pydantic 验证之前，用 `setdefault()` 为所有必填字段提供合理的兜底值（如日文名默认用英文名、category 默认 "sightseeing"）。**通用原则**: 永远不要假设 AI 会返回完整的 JSON，即使 prompt 中明确要求了 schema。
- **Next.js rewrites 有隐藏超时**: Next.js 的 `rewrites()` 代理有默认超时限制，对于 AI 生成这种 60-120 秒的长请求会直接返回 500。**解决方案**: 使用 Route Handler (`app/api/.../route.ts`) 手动代理，可以自定义 `AbortSignal.timeout(180_000)`。

### v1.2 (2026-03-15) — 数据质量提升

**变更内容：**
- 添加景点参考照片（photo_url）和特色亮点（highlight）字段
- AI prompt 增加"已关闭场所黑名单"防止推荐已停业景点
- 增加"按区域聚类"规划原则，每天景点集中在同一区域，减少无效移动
- 增强活动描述质量要求（感官细节、特色卖点）

**经验教训：**
- **已关闭场所问题**: AI 模型的训练数据可能过时，会推荐已永久关闭的场所（如大江户温泉物语台场店 2021 年已关闭）。**解决方案**: 在 system prompt 中维护"已知关闭场所黑名单"，并指示 AI 优先选择知名度高、经营稳定的场所。**未来改进**: 接入 Google Places API 实时验证场所营业状态。
- **行程规划散乱问题**: 未加限制时 AI 会在同一天安排跨城区的景点（如浅草→涩谷→上野→六本木），导致大量无效移动。**解决方案**: 在 prompt 中明确要求按区域聚类，给出正面/反面示例，并列出各城市的区域分组。

### v1.1 (2026-03-15) — 实用信息增强

**变更内容：**
- 每个景点添加 Google Maps 链接（一键导航）
- 显示营业时间和预约电话
- 景点之间显示交通方式、时间、费用（步行/地铁/公交等）
- axios 和 Vite proxy 超时时间设为 3 分钟（适配 Kimi API 响应时间）

**经验教训：**
- **API 超时问题**: Kimi API 生成长行程时响应时间可能超过 60 秒，默认超时会导致请求失败。**解决方案**: axios timeout 和 Vite proxy timeout 统一设为 180 秒。
- **Vite HMR 状态丢失**: 开发时编辑文件会触发热更新，重置 React 状态导致正在进行的 API 请求丢失。这不是 bug，但在演示时需注意。

### v1.0 (2026-03-15) — 初始 MVP + API 切换

**变更内容：**
- 从 Gemini API 切换到 Kimi (Moonshot) API（Gemini 免费额度用尽）
- 创建统一 AI 服务层（ai_service.py），支持多 provider 切换
- 添加自动重试和指数退避（429 限流处理）

**经验教训：**
- **API 配额问题**: Gemini 免费额度每日有限，高频测试会快速耗尽。**解决方案**: 支持多 API provider 切换，在 .env 中配置即可切换。
- **异步兼容性**: Kimi 的 OpenAI SDK 异步客户端在 FastAPI 中有兼容问题。**解决方案**: 使用 `asyncio.to_thread()` 包装同步调用。
- **.env 覆盖 config 默认值**: pydantic-settings 的 .env 文件会覆盖代码中的默认值，修改代码默认值不生效。**解决方案**: 同时修改 .env 文件和 config.py。
